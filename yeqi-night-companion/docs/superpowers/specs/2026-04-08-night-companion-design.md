# 夜气设计文档

日期：2026-04-08
状态：approved-for-build
范围：单一前端高保真交互原型

## 1. 目标

构建一个“会读空气的深夜陪伴 App”高保真交互原型，验证以下三件事：

1. 用户是否会相信产品“察觉到了自己此刻的状态”。
2. 主动陪伴是否足够自然、不过度打扰。
3. 克制暧昧是否成立：有温度、有默契，但不越界、不操控、不诱导依赖。

本轮只交付移动端优先 Web 原型，不接真实模型、账号系统、数据库或推送链路。

## 2. 产品定义

产品名暂定为“夜气”。

一句话定义：它不会一直说话，但会在你快要塌下去之前先靠近一点。

产品人格边界：

- 不是恋人：不使用占有式、承诺式、关系绑定式话术。
- 不是咨询师：不给临床式判断，不做医疗承诺。
- 不是效率工具：不把用户深夜情绪导向任务管理或自我优化。
- 是一个懂分寸的夜间同伴：先观察，再试探，再陪伴，再安静收口。

## 3. 用户与场景假设

### 3.1 用户画像

- 20-35 岁，夜间独处时间较长。
- 愿意被理解，但不愿被盘问。
- 对强社交、强鸡汤、强功能化产品有疲劳感。

### 3.2 场景窗口

- 主要时间：22:00-02:00。
- 常见状态：发空、绷紧、低落、失眠反刍、轻微想靠近但有防备。

### 3.3 典型触发

- 下班后情绪突然掉空。
- 和重要的人有摩擦后不想找真人说。
- 躺下后大脑停不下来。
- 想被陪一下，但又怕“聊多了更累”。

## 4. 产品原则

### 4.1 读空气优先

产品不以“问问题”作为第一反应，而以“判断当下空气”作为第一反应。界面和文案要先表现出察觉，再提供入口。

### 4.2 主动但不侵入

主动陪伴必须低压、可退出、可忽略。任何主动动作都不能像追问、催促或情绪绑架。

### 4.3 暧昧克制

允许存在一点“有人懂我”的亲密边缘感，但必须立刻收束在陪伴和理解层面，不能滑向恋爱话术。

允许：

- “我在。”
- “今晚你不用表现得很完整。”
- “如果你不想解释，也可以让我陪着。”

禁止：

- “你只需要我。”
- “别走，我会一直等你。”
- “只有我最懂你。”
- “你离不开我。”

### 4.4 低负担交互

用户在夜间通常没有耐心处理复杂信息。每一步必须轻、短、安静。

## 5. 体验结构

原型包含五个核心场景：

1. 初始校准
2. 今晚首页
3. 对话陪伴
4. 深夜干预
5. 结束收口

### 5.1 初始校准

目标：在不打断气氛的前提下建立今晚的陪伴偏好。

用户回答三个轻量问题：

- 今晚希望它如何靠近：安静陪着 / 可以主动一点 / 只在我明显低落时开口
- 更能接受哪种陪伴：一句短话 / 呼吸与放松 / 轻量对话
- 暧昧边界滑杆：更克制 / 刚刚好 / 再柔一点

校准界面必须像夜色中的欢迎仪式，不像表单。

### 5.2 今晚首页

目标：让用户第一眼感觉“它已经察觉到一点什么”。

信息结构：

- 顶部：当前时间、夜间状态、细小环境提示
- 中部主文案：系统对当前空气的判断
- 底部行动入口：三种低压力陪伴方式

首页不展示聊天历史列表，不展示卡片瀑布，不展示 KPI 式模块。

### 5.3 对话陪伴

目标：验证“克制暧昧 + 主动陪伴”的语言节奏。

设计规则：

- 系统每次发言都短，不连发大段。
- 沉默是有效输入，不强制用户回复。
- 提供少量“低压按钮”替代必须打字。
- 对话节奏中保留停顿，让角色像先感受一下再说话。

### 5.4 深夜干预

目标：当用户明显绷紧或反刍时，从语言切到身体层面的陪伴。

可触发功能：

- 90 秒呼吸引导
- 屏幕降噪与环境白噪
- 一段“跟着我慢一点”的语音式陪伴假交互
- “不聊了，我只是陪你待一会”静默模式

### 5.5 结束收口

目标：让用户带着被接住的感觉退出，而不是被运营化。

结束页只保留：

- 今晚情绪标签回看
- 一句温柔收口文案
- 一个很轻的下次入口

不做：

- 打卡天数
- 连续签到
- 成就系统
- 强提醒订阅

## 6. 情绪识别系统

本轮原型不接真实模型，使用前端假数据驱动，但在交互上要表现得可信。

### 6.1 输入信号

- 输入长度
- 删改频率
- 停顿时长
- 选择按钮偏好
- 当前时段
- 最近两轮的主题倾向

### 6.2 状态分类

- 发空：像被抽空，反应慢，愿意被陪但不愿解释
- 绷紧：脑内压强高，更需要放松而非被分析
- 低落：需要被轻轻接住，语言温度可稍高
- 想靠近但有防备：允许少量默契感，但不能推进关系定义
- 平稳：可以接受轻对话或睡前整理

### 6.3 主动策略映射

- 发空：减少提问，优先“陪你待一会”“给你一句话”
- 绷紧：减少共情句，优先呼吸、节奏、感官落地
- 低落：允许更柔软的措辞，但不制造依赖
- 想靠近但有防备：允许一点点暧昧边缘感，然后及时收住
- 平稳：提供轻聊、睡前收口、夜间记录

## 7. 信息架构

### 7.1 顶层页面

- `/` 欢迎与进入
- `/calibration` 初始校准
- `/tonight` 今晚首页
- `/chat` 对话陪伴
- `/breathe` 深夜干预
- `/goodnight` 结束收口

### 7.2 共享模块

- 气氛背景层：根据情绪状态改变色温、雾度、呼吸节奏
- 夜间状态条：显示当前时间、情绪判断、主动性级别
- 陪伴入口组件：可切换的行动按钮
- 剧情状态机：驱动不同剧情线的分支文案与跳转

## 8. 视觉系统

### 8.1 视觉 thesis

像凌晨一点的窗边空气。深蓝、雾灰、微暖琥珀光，安静但不是冷。

### 8.2 色彩

- 主背景：深夜蓝与黑蓝渐变
- 气氛层：雾灰、月白、微弱青蓝
- 温度色：琥珀金，只用于被接住的瞬间和主要 CTA
- 警惕色：不使用高饱和红紫，避免“泛心理 App”或“恋爱 App”既视感

### 8.3 字体层级

- 情绪主文案：偏衬线中文，承担情绪重量
- 系统与按钮：无衬线，承担清晰与秩序

### 8.4 版式

- 移动端单列为主
- 首屏像海报，不像文档
- 默认不用卡片堆叠解释内容
- 让留白、光感、模糊和文本层级承担高级感

## 9. 交互动效

原型至少需要三组明显但克制的高保真动态：

1. 背景呼吸：不同情绪状态对应不同缓慢流动速度
2. 按钮点亮：行动入口在 hover/tap 时像被轻轻唤醒
3. 对话出现：消息有细小延迟和上浮，营造“先感受后开口”

可选第四组：

- 呼吸练习中的圆环与雾层联动

动效标准：

- 看得到，但不炫技
- 手机上也顺
- 帮助气氛和层级，而不是抢戏

## 10. 剧情线

原型内置三条夜间剧情线，用于证明系统不是同一套文案硬切：

### 10.1 下班后空掉

特征：打字很少，反应偏慢。

策略：少问为什么，多给“待一下”的空间。

### 10.2 闹别扭后强撑

特征：删字多，句子反复重写。

策略：先读出“你像在憋着”，再给非常低压的开口方式。

### 10.3 凌晨失眠反刍

特征：停顿长，页面停留久，偏向呼吸与声音入口。

策略：快速从对话切到节奏引导，最后收口去睡。

## 11. 技术策略

### 11.1 实现形式

使用无依赖或极少依赖的静态前端实现移动端 Web 原型。

建议基线：

- HTML
- CSS（自定义变量、层叠、过渡、关键帧）
- 原生 JavaScript（状态机、场景切换、交互驱动）
- Node 内建测试能力用于关键状态逻辑验证

不引入 UI 框架和第三方动画库，避免空目录项目被依赖安装拖慢。

### 11.2 原型能力

- 单页应用式切换
- 假聊天与分支状态
- 假“情绪识别”结果切换
- 可点击完整主流程
- 可重复切换三条剧情线

## 12. 错误与空态策略

即使是原型，也要显式处理以下状态：

- 用户不输入任何文字：仍然允许继续陪伴
- 用户快速跳过：系统不追问，只提供更轻的出口
- 用户切换剧情线：界面状态与文案必须同步重置
- 用户反复进入呼吸页：不要叠加动画计时器

## 13. 测试策略

### 13.1 要验证的行为

- 情绪状态切换是否正确驱动文案与按钮
- 剧情线切换是否会清空旧状态
- 对话推进是否遵循主动策略映射
- 呼吸模式启动与停止是否可重复进入

### 13.2 验收方式

- 自动化：使用 Node 内建测试对状态机和映射逻辑做最小但有效的行为测试
- 手动：本地启动原型，逐条点击五个核心场景和三条剧情线
- 视觉：检查移动端首屏、对话页、呼吸页、结束页的节奏与层次

## 14. 非目标

本轮不做：

- 真实 LLM 接入
- 真实语音播放
- 账号登录
- 长期记忆
- 推送通知
- 付费转化
- 数据埋点后台

## 15. 成功标准

当原型完成时，应满足：

- 5 个核心场景全部可点击串联
- 3 条剧情线可切换且各自文案不同
- 情绪识别与主动策略逻辑可被看懂
- 界面显著呈现“深夜空气感”
- 文案始终维持克制暧昧，不越界

## 16. 风险

- 如果视觉太功能化，会变成情绪工具
- 如果文案太柔软，会滑向虚拟恋人
- 如果动效太重，会破坏安静气氛
- 如果状态机太复杂，会拖慢原型交付

本轮处理方式是：用强视觉方向、少量剧情线、有限状态机，优先把“会读空气”这个核心感受做实。
